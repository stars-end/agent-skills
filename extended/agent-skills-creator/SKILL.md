# Agent Skills Creator

Create or update canonical skills in `~/agent-skills` with baseline-safe repo workflow.

## Goal

Skill work should produce:
- the right skill in the right category
- clear trigger metadata
- minimal, current instructions
- optional `agents/openai.yaml` metadata when the skill should surface in compatible UIs
- regenerated `AGENTS.md` / baseline artifacts via `make publish-baseline`

## Canonical Contract

- `~/agent-skills` is the source of truth for shared skills.
- Use a worktree for any skill change.
- Track the work in canonical Beads (`~/bd`) before making substantive edits.
- Do not teach stale `.claude/skills`, `skill-rules.json`, repo-local `.beads`, or legacy `bd sync` flows unless explicitly documenting compatibility.
- Regenerate baseline after any skill addition, removal, rename, deprecation, or metadata change.

## When To Use

Use this skill when the user asks to:
- create a new skill
- improve an existing skill
- deprecate or replace a skill
- align a skill with current `agent-skills` practice
- update `AGENTS.md` skill indexing indirectly via baseline regeneration

## Skill Placement Rules

Choose the smallest correct namespace:
- `core/` for default workflow skills
- `extended/` for advanced or optional workflows
- `health/` for diagnostics and recovery
- `infra/` for infrastructure or fleet operations
- `dispatch/` for delegation / runner surfaces
- `railway/` for Railway-specific skills
- `safety/` for protective guardrails

Use the patterns in:
- `resources/skill-layout.md`
- `resources/deprecation-pattern.md`

## Workflow

### 1. Establish the Work Item

If there is no Beads issue yet:
- create one in `~/bd`
- use a meaningful title and issue type

Then create a worktree:

```bash
cd ~/agent-skills
dx-worktree create <beads-id> agent-skills
```

### 2. Decide the Change Type

Choose one:
- new canonical skill
- update existing skill
- deprecate and redirect old skill
- split one skill into two

Prefer the smallest change that improves activation and execution quality.

### 3. Author the Skill

Create or update:
- `SKILL.md`
- optional `resources/` files only when they reduce main-file noise
- optional `agents/openai.yaml` when the skill should appear in compatible skill UIs

Keep the main file focused on:
- what the skill is for
- when to use it
- the active workflow
- the active contract

Do not keep dead historical guidance in the main skill.

### 4. Deprecation Pattern

If replacing an old skill:
- keep the old skill name active long enough to catch trigger phrases
- mark it as deprecated in the description/body
- redirect to the new canonical skill
- keep only minimal compatibility text

Use this when removing ambiguity is more important than preserving old detail.

### 5. Regenerate Baseline

From the `agent-skills` worktree:

```bash
make publish-baseline
```

This is mandatory after:
- adding a skill
- removing a skill
- renaming a skill
- changing trigger metadata
- changing UI metadata

### 6. Sanity Check

Check:
- new files exist
- descriptions reflect actual trigger conditions
- no stale repo-local Beads or old skill-system assumptions were introduced
- `AGENTS.md` and `dist/*` were regenerated by the baseline task

### 7. Commit and PR

Commit with:
- `Feature-Key: bd-...`
- `Agent: <id>`

Open a PR for human review when the change is complete.

## Creation Rules

- Prefer concise skills over encyclopedic ones.
- Create `resources/` only when it materially improves readability.
- Reuse established patterns already present in `~/agent-skills`.
- If the user repeatedly asks for a workflow, create a dedicated skill instead of overloading a generic meta-skill.

## Deprecation Rules

- Deprecate by redirecting, not by silently deleting.
- Keep the deprecated skill good enough to route the user correctly.
- Remove stale examples that teach the wrong runtime or file layout.

## Anti-Patterns

- teaching `.claude/skills/...` as the primary path
- mentioning nonexistent helper scripts as mandatory
- forgetting `make publish-baseline`
- editing canonical `~/agent-skills` directly instead of a worktree
- letting `AGENTS.md` drift after skill edits

## Deliverable Contract

Return:
- `SKILLS_ADDED`
- `SKILLS_UPDATED`
- `SKILLS_DEPRECATED`
- `BASELINE_REGENERATED: yes|no`
- `PR_URL`
